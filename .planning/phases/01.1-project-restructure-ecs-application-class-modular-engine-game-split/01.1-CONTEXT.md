# Phase 1.1: Project Restructure — Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Restructure the flat project into a modular engine/game architecture with ECS (EnTT), an Application class that owns the game loop and subsystems, an event bus for cross-system communication, and a SceneManager for level transitions. All existing functionality must continue working after the restructure — this is a refactoring phase, not a feature phase.

</domain>

<decisions>
## Implementation Decisions

### Source Tree Organization
- **D-01:** Engine/game split: `src/engine/` for reusable engine code, `src/game/` for game-specific code
- **D-02:** Engine subdirectories: core/, ecs/, rendering/, input/, scene/, ui/ (plus future: physics/, audio/)
- **D-03:** Game subdirectories: scenes/, components/, systems/, combat/ (future), enemies/ (future)

### Application Architecture
- **D-04:** Application class owns the game loop, subsystem lifecycle. main.cpp becomes ~5 lines.
- **D-05:** Systems register via `app.addSystem<T>()` pattern
- **D-06:** Application provides deltaTime, window handle, and ECS registry to systems

### ECS (EnTT)
- **D-07:** Adopt EnTT for all game objects — entities with components, systems process them
- **D-08:** Camera becomes CameraComponent + CameraSystem (ECS-driven)
- **D-09:** Lights become entities with LightComponent
- **D-10:** Render objects become entities with TransformComponent + MeshComponent

### Communication
- **D-11:** Event bus for cross-system communication (DamageEvent, InputEvent, etc.)
- **D-12:** Systems publish/subscribe to events — decoupled, no direct system-to-system calls

### Scene Management
- **D-13:** SceneManager with Scene base class
- **D-14:** Scenes push/pop (MainMenu, GameLevel, PauseMenu)
- **D-15:** Each scene owns its entity setup — CathedralScene spawns entities into the registry

### Build System
- **D-16:** Static libs per module in CMake (engine_core, engine_rendering, engine_input, etc.)
- **D-17:** Strict module boundaries — each lib declares its own include dirs and dependencies

### Assets
- **D-18:** Assets organized by type: assets/shaders/, assets/models/, assets/sounds/

### Claude's Discretion
- Exact EnTT integration patterns (registry ownership, system base class design)
- Event bus implementation details (template-based, type-erased, etc.)
- CMake target naming conventions
- Header include structure within modules

</decisions>
