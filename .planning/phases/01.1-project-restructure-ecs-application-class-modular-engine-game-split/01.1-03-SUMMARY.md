**Five concrete ECS systems replacing main.cpp inline logic: Scene/SceneManager stack, InputSystem (GLFW+ImGui wrapper), CameraSystem (exact yaw/pitch camera math), RenderSystem (FBO/dither/ImGui pipeline) -- all implementing the System base class**

## Accomplishments

- Scene base class defines onEnter/onExit/onUpdate lifecycle used by future game scenes (CathedralScene migration in plan 04)
- SceneManager implements push/pop stack model allowing overlay scenes (pause menu on top of game scene)
- InputSystem wraps GLFW key queries behind System interface, provides wantsCaptureMouse() for ImGui gate
- CameraSystem extracts camera logic from main.cpp with byte-for-byte identical math: cos(radians(yaw))*cos(radians(pitch)), sin(radians(pitch)), sin(radians(yaw))*cos(radians(pitch))
- RenderSystem extracts the full render pipeline: FBO bind, drawScene, dither apply, ImGui overlay, resolution change, F12/auto screenshot -- identical sequence to main.cpp
- CMakeLists.txt updated with engine_scene, engine_input libs and game lib expanded with CameraSystem/RenderSystem sources

## Decisions Made

- CameraSystem receives `InputSystem&` by constructor injection rather than looking it up -- avoids application-level coupling and makes dependency explicit
- Window aspect ratio used for projection matrix (not FBO aspect) -- FBO at 720p defaults equals 1280x720 which matches window, so aspect is identical to main.cpp
- RenderSystem builds RenderObject/PointLight vectors fresh every frame from ECS -- simple and correct; caching would require change tracking
- `constexpr static int RES_W[]/RES_H[]` in header with out-of-line definitions in .cpp -- C++20 handles this cleanly
