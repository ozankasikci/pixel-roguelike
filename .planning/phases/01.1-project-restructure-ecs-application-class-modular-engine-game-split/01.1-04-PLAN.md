    **src/game/scenes/CathedralScene.cpp** -- Refactor build() into onEnter():
    The geometry generation stays identical. The difference is HOW objects are registered:

    OLD pattern (vector-based):
    ```cpp
    objects_.push_back({meshPtr, modelMatrix});
    lights_.push_back({position, color, radius, intensity});
    ```

    NEW pattern (entity-based):
    ```cpp
    auto entity = app.registry().create();
    app.registry().emplace<MeshComponent>(entity, MeshComponent{meshPtr, modelMatrix, true});
    // useModelOverride=true because CathedralScene pre-computes model matrices with custom transforms
    entities_.push_back(entity);
    ```

    For lights:
    ```cpp
    auto lightEntity = app.registry().create();
    app.registry().emplace<TransformComponent>(lightEntity, TransformComponent{position, {}, {1.0f}});
    app.registry().emplace<LightComponent>(lightEntity, LightComponent{color, radius, intensity});
    entities_.push_back(lightEntity);
    ```

    For the camera:
    ```cpp
    auto cameraEntity = app.registry().create();
    app.registry().emplace<TransformComponent>(cameraEntity, TransformComponent{{0.0f, 2.0f, 5.0f}, {}, {1.0f}});
    app.registry().emplace<CameraComponent>(cameraEntity);  // defaults: yaw=-90, pitch=0, fov=70
    entities_.push_back(cameraEntity);
    ```
    Camera initial position (0, 2, 5) matches the current scene.cameraPos() = glm::vec3(0.0f, 2.0f, 5.0f) from main.cpp line 59.

    CRITICAL: All geometry generation code (createCylinder, addArch, floor, ceiling, walls, pillars, arches, floor tiles, steps) stays EXACTLY the same. Only the way results are stored changes from vectors to ECS entities. The mesh unique_ptrs remain in meshes_ for memory management.

    For the static helper functions (createCylinder, addArch): Keep them as-is. They create Mesh objects. The only change is that instead of pushing to objects_ vector, we create entities. The addArch helper needs to be updated to accept registry and entities vector instead of objects vector:

    Per D-04: main.cpp becomes minimal. Replace the entire 210-line god function with approximately 15-25 lines.

    The new main.cpp:
    ```cpp
    #include "engine/core/Application.h"
    #include "engine/scene/SceneManager.h"
    #include "engine/input/InputSystem.h"
    #include "game/systems/CameraSystem.h"
    #include "game/systems/RenderSystem.h"
    #include "game/scenes/CathedralScene.h"

    #include <spdlog/spdlog.h>
    #include <memory>
    #include <string>

    int main(int argc, char* argv[]) {
        spdlog::set_level(spdlog::level::info);

        // Parse --screenshot path for automated capture
        std::string autoScreenshotPath;
        for (int i = 1; i < argc; ++i) {
            if (std::string(argv[i]) == "--screenshot" && i + 1 < argc) {
                autoScreenshotPath = argv[i + 1];
                ++i;
            }
        }

        Application app(1280, 720, "Pixel Roguelike");

        // Register systems (per D-05: app.addSystem pattern)
        auto& input = app.addSystem<InputSystem>();
        auto& camera = app.addSystem<CameraSystem>(input);
        auto& render = app.addSystem<RenderSystem>();

        if (!autoScreenshotPath.empty()) {
            render.enableAutoScreenshot(autoScreenshotPath, 10);
        }

        // Push the cathedral scene (per D-14, D-15)
        SceneManager sceneManager;
        sceneManager.pushScene(std::make_unique<CathedralScene>(), app);

        // Run the game loop (per D-04)
        app.run();

        sceneManager.popScene(app);

        spdlog::info("Shutting down");
        return 0;
    }
    ```

    NOTE: The SceneManager is created in main for now. In a future phase, Application could own it. But for this restructure phase, keeping it simple and explicit is the right call.

    IMPORTANT: The --screenshot command-line parsing must be preserved exactly as it is now, since automated testing depends on it. The auto-screenshot gets wired to RenderSystem via enableAutoScreenshot().

    IMPORTANT: The Application::run() method handles the main loop. The SceneManager's active scene gets updated through the systems (RenderSystem queries entities that CathedralScene created). The scene lifecycle is: push scene -> onEnter spawns entities -> systems process entities each frame -> popScene calls onExit to cleanup.

    IMPORTANT: The system registration order matters:
    1. InputSystem first (updates key state)
    2. CameraSystem second (reads input, updates camera matrices)
    3. RenderSystem third (reads camera matrices and entity data, renders)
    This matches the current main.cpp's implicit order: poll events, process input, compute camera, render, ImGui.
