## User Constraints (from CONTEXT.md)

### Locked Decisions
### Inventory Scope
- **D-01:** This phase covers weapons only. Other carryable item categories are deferred.
- **D-02:** Weapons are inventory items that can be owned independently of being equipped.
- **D-03:** Non-weapon carryables, key items, and broader progression inventory remain out of scope for this phase.

### Equipment Model
- **D-04:** Use an explicit hand-based equipment model inspired by Souls games rather than a single active weapon slot.
- **D-05:** The player has one right-hand weapon slot and one left-hand weapon slot.
- **D-06:** Two-handed weapons are supported as a weapon type and should occupy the hand-equipment model appropriately rather than being treated as a separate unrelated system.
- **D-07:** Weapon changes happen from the inventory screen only for now; in-world quick cycling is deferred.

### Burden / Carry Weight
- **D-08:** Use true equip-load style burden for this phase: equipped weapons determine burden.
- **D-09:** Unequipped weapons in the inventory do not add burden.
- **D-10:** No separate slot-grid or bag-cap inventory model should be introduced in this phase.

### Inventory Presentation
- **D-11:** The inventory is a dedicated paused screen, not a live overlay while gameplay continues.
- **D-12:** The presentation should feel Dark Souls-inspired: category-driven, list-based, and utilitarian rather than grid-based.
- **D-13:** The paused screen should include a category list, item list, detail pane, and a visible burden/equip-load meter.
- **D-14:** The default key to open the inventory is `I`.

### Claude's Discretion
- Exact category names and menu wording, as long as the UI remains Souls-like and minimal
- Exact visual composition, spacing, typography, and 1-bit treatment of the paused inventory screen
- Exact behavior of how a two-handed weapon displaces or locks left/right-hand equipment, as long as it stays consistent with the chosen hand-based model
- Internal data structures for inventory entries, equipment state, and burden calculation

### Deferred Ideas (OUT OF SCOPE)
- General non-weapon inventory management
- Key/progression item handling
- In-world quick cycling of equipped weapon slots
- Broader RPG encumbrance/stat systems beyond equipped-weapon burden

## Summary

This phase should be planned as a gameplay-state extension, not as a generic inventory framework. The current repo already has the right primitives: run-scoped persistence in `RunSession`, content-authored weapon definitions in `ContentRegistry`, ECS registry context for transient UI state, and centralized input/cursor ownership in `InputSystem`. The shortest stable path is to extend those systems rather than inventing a separate inventory subsystem or data store.

The critical modeling decision is to separate owned weapons from equipped hands. The existing `RunSession::LoadoutSlot { definitionId, equipped }` in `src/game/session/RunSession.h` is too weak for left-hand/right-hand/two-hand semantics and future combat integration. Plan for an explicit equipment state with right-hand occupancy, left-hand occupancy, and a consistent rule for two-handed weapons. Burden should be derived from the effective equipped state every frame or on demand, not stored independently.

The critical interaction decision is to treat the inventory as a real paused state. The current codebase already gates movement and interaction on cursor lock and `PlayerInteractionLockComponent`; the inventory should reuse that pattern so opening the menu on `I` unlocks the cursor, suppresses camera/movement/interaction, and closes cleanly back to locked first-person control. The UI should stay list-based and utilitarian, and for planning purposes should assume the current `Dear ImGui` path is the fastest way to ship the first version while keeping the visual treatment intentionally minimal and high-contrast.

**Primary recommendation:** Extend `RunSession` and `WeaponDefinition`, add a dedicated `InventorySystem` plus transient menu state in registry context, and make burden a pure function of effective hand occupancy.

## Architecture Patterns

### Recommended Project Structure
```text
src/game/
├── content/                 # extend WeaponDefinition metadata and parser
├── session/                 # extend RunSession with owned/equipped weapon state
├── systems/                 # add InventorySystem / equipment application logic
├── ui/                      # add InventoryMenuState and related transient UI state
└── components/              # only add ECS components if combat/viewmodel bridge truly needs them
```

### Pattern 1: Separate Owned Inventory from Equipped Hands
**What:** Store owned weapon entries independently from the player's active hand state.
**When to use:** Always. This is the only model that cleanly satisfies left hand, right hand, two-handed weapons, and inventory-owned-but-unequipped weapons.
**Example:**
```cpp
struct OwnedWeaponEntry {
    std::string weaponId;
};

enum class HandOccupancy {
    Empty,
    OneHandedWeapon,
    TwoHandedWeapon
};

struct EquipmentState {
    std::optional<std::string> rightHandWeaponId;
    std::optional<std::string> leftHandWeaponId;
    std::optional<std::string> twoHandedWeaponId;
};
```
Source: repo architecture in `RunSession` + `WeaponDefinition`; explicit hand state is an inference from the locked phase decisions.

### Pattern 2: Persistent Gameplay State in `RunSession`, Transient Menu State in `registry.ctx()`
**What:** Owned/equipped weapons belong in `RunSession`; menu open/closed state, selected category, highlighted item, and scroll position belong in registry context.
**When to use:** For all paused inventory UI state that should reset when the level session is rebuilt.
**Example:**
```cpp
struct InventoryMenuState {
    bool open = false;
    int selectedCategory = 0;
    int selectedItem = 0;
};

void ensureInventoryMenuState(entt::registry& registry) {
    auto& ctx = registry.ctx();
    if (!ctx.contains<InventoryMenuState>()) {
        ctx.emplace<InventoryMenuState>();
    }
}
```
Source: EnTT registry context docs and existing repo use of `InteractionPromptState` / `MeshAssetProvider`.

### Pattern 3: Burden Is Derived from Effective Equipment, Never Authored Twice
**What:** Compute burden from the weapon definitions currently occupying the effective hands.
**When to use:** Every time the UI needs burden, and every time equip state changes.
**Example:**
```cpp
float computeEquipBurden(const EquipmentState& equipment, const ContentRegistry& content) {
    float total = 0.0f;

    if (equipment.twoHandedWeaponId.has_value()) {
        if (const auto* weapon = content.findWeapon(*equipment.twoHandedWeaponId)) {
            total += weapon->equipWeight;
        }
        return total;
    }

    if (equipment.rightHandWeaponId.has_value()) {
        if (const auto* weapon = content.findWeapon(*equipment.rightHandWeaponId)) {
            total += weapon->equipWeight;
        }
    }
    if (equipment.leftHandWeaponId.has_value()) {
        if (const auto* weapon = content.findWeapon(*equipment.leftHandWeaponId)) {
            total += weapon->equipWeight;
        }
    }

    return total;
}
```
Source: phase decision D-08/D-09 and local code constraints; burden derivation is the recommended invariant.

### Pattern 4: Inventory Open/Close Owns Cursor and Interaction Gating
**What:** Toggling inventory should both unlock/lock the cursor and mark the player interaction lock active while the menu is open.
**When to use:** On `I` open/close and any forced close path.
**Example:**
```cpp
if (input.isKeyJustPressed(GLFW_KEY_I)) {
    menu.open = !menu.open;
    if (menu.open) {
        input.unlockCursor();
        playerLock.active = true;
    } else {
        input.lockCursor();
        playerLock.active = false;
    }
}
```
Source: existing `InputSystem`, `PlayerInteractionLockComponent`, `PlayerMovementSystem`, `CameraSystem`, and `InteractionSystem`.

### Anti-Patterns to Avoid
- **Single `equipped` bool on inventory entries:** Cannot represent right-hand vs left-hand vs two-handed occupancy.
- **General-purpose bag/grid system:** Directly contradicts D-10 and adds UI/logic complexity unrelated to this phase.
- **Storing burden as independent mutable session data:** Leads to drift when equip state changes.
- **Using `ItemDefinition` for weapons:** The repo already has a separate `WeaponDefinition`; collapsing them now will make combat integration worse.
- **Making inventory a visual overlay while gameplay continues:** Conflicts with D-11 and will create cursor/movement/input bugs in the current architecture.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Paused inventory UI | A new retained-mode widget toolkit | Existing Dear ImGui integration | Already wired to GLFW/OpenGL and fast to iterate |
| Equipment state | A boolean flag per owned item | Explicit hand/equipment state | Two-handed and left/right semantics require it |
| Burden tracking | A cached mutable stat with manual sync | Pure calculation from equipment + weapon metadata | Eliminates sync bugs |
| Inventory persistence | A second persistence store outside `RunSession` | Extend `RunSession` | Future save/load already wants run-scoped state |
| Large text/menu system | A custom text-entry framework | Existing list/selectable primitives and static labels | This phase is selection-driven, not text-input-driven |

**Key insight:** The deceptively hard part here is not drawing a menu, it is keeping ownership, equipment occupancy, burden, and pause/input state perfectly consistent. Reuse the current engine paths and keep the model explicit.

## Common Pitfalls

### Pitfall 1: Trying to Reuse `RunSession::LoadoutSlot` As-Is
**What goes wrong:** A single `equipped` boolean cannot express which hand holds a weapon or whether it occupies both hands.
**Why it happens:** The current `RunSession` model was sufficient for a single loadout concept but not for Souls-like equipment.
**How to avoid:** Replace or supersede `LoadoutSlot` with separate owned inventory and explicit equipment state.
**Warning signs:** Equip actions require special-case booleans like `equippedLeft`, `equippedRight`, `twoHandedEquipped`, and burden logic starts branching on item position in a vector.

### Pitfall 2: Treating Pause as Only a UI Concern
**What goes wrong:** Camera or interaction continues running while the inventory is open.
**Why it happens:** Current systems gate on both cursor lock and `PlayerInteractionLockComponent`; only toggling one path is insufficient.
**How to avoid:** Make inventory open/close update cursor mode and player lock state together.
**Warning signs:** Mouse look still moves in the menu, `E` prompts remain active, or the player drifts while browsing inventory.

### Pitfall 3: Double-Counting Two-Handed Weapon Weight
**What goes wrong:** Burden becomes incorrect because a two-handed weapon is counted once per hand.
**Why it happens:** The planner models hand occupancy as two filled slots without defining the burden invariant.
**How to avoid:** Define an effective-equipment view where a two-handed weapon is one equipped weapon occupying both hands.
**Warning signs:** Burden jumps to exactly 2x the weapon's authored weight when equipping a two-handed weapon.

### Pitfall 4: Under-Specifying Weapon Content Metadata
**What goes wrong:** Systems start hardcoding weight/handedness/category because content definitions cannot express them.
**Why it happens:** `WeaponDefinition` currently only has `id`, `displayName`, `slot`, `meshId`, `damage`, `range`, and `cooldown`.
**How to avoid:** Add authored metadata now: at minimum handedness, equip weight, category, and UI description text.
**Warning signs:** Switch statements by weapon id appear in UI or burden code.

### Pitfall 5: Forgetting the Current Asset Parser Is Whitespace-Tokenized
**What goes wrong:** Multi-word names or descriptions fail to parse.
**Why it happens:** `ContentRegistry.cpp` tokenizes definition lines by whitespace, and existing assets encode spaces with underscores.
**How to avoid:** Either keep the underscore convention for this phase or explicitly extend the parser before planning richer text fields.
**Warning signs:** Inventory labels truncate after the first word, or parsing starts failing on new weapon definitions.

### Pitfall 6: Binding Future Combat Too Tightly to Menu Implementation
**What goes wrong:** Combat later depends on menu-only state rather than stable equipment queries.
**Why it happens:** The inserted phase sits immediately before combat, so it is tempting to let combat read raw UI state.
**How to avoid:** Expose stable functions like "effective right-hand weapon" and "effective left-hand weapon" from session/equipment state.
**Warning signs:** Combat systems include `InventoryMenuState.h` or read selected menu rows directly.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Single active-weapon/loadout flag | Explicit owned inventory plus explicit equipment occupancy | Current standard in action-RPG equipment design; required by this phase's locked decisions | Supports left/right/two-handed rules cleanly |
| Grid/bag inventory | Category + list + detail pane | Long-established genre convention for Souls-like menus | Better fit for 1-bit readability and weapon-only scope |
| Mutable cached equip-load stat | Derived burden from equipped items | Standard data-modeling practice | Lower bug surface and easier testing |
| Custom UI framework for every gameplay screen | Reuse existing engine UI layer first | Current repo architecture | Faster insertion phase with lower integration risk |

**Deprecated/outdated:**
- Treating all carried weapons as contributing to weight: explicitly rejected by D-08 and D-09.
- Treating inventory as "all-purpose item management": explicitly out of scope for this phase.

## Open Questions

1. **What exact two-handed rule should the planner lock in?**
   - What we know: Two-handed weapons must be supported and must occupy the hand-equipment model consistently.
   - What's unclear: Whether equipping a two-handed weapon clears both one-hand slots, stores previous offhand temporarily, or blocks equip until the offhand is manually cleared.
   - Recommendation: Pick one strict invariant for v1. The cleanest plan is "two-handed weapon occupies both hands and clears conflicting one-hand occupancy immediately."

2. **Should the first shipped inventory screen be ImGui-backed or custom-rendered?**
   - What we know: Existing ImGui integration is already functional and fastest.
   - What's unclear: Whether the team wants this phase to pay the cost of a fully custom 1-bit menu renderer now.
   - Recommendation: Plan ImGui for this phase, with the visual treatment constrained to a stark list/detail layout and a later migration path if Phase 4 needs bespoke UI.

3. **Where should viewmodel ownership land once combat arrives?**
   - What we know: `LevelLoader` currently spawns a single `hand` viewmodel entity, and combat will need the equipped weapon to drive what appears in first person.
   - What's unclear: Whether this phase should already swap meshes or only establish the data/query seam.
   - Recommendation: Keep this phase focused on data and paused UI. Expose equipment queries so Phase 3 can own actual combat/viewmodel behavior.
