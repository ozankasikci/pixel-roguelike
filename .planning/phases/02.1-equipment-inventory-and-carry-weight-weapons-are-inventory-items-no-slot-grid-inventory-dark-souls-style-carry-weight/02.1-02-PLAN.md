  <action>
    Implement the gameplay-facing pause/inventory controller for D-01 through D-14 using the existing ECS/system model instead of inventing a separate framework.

    Create `src/game/ui/InventoryMenuState.h` as registry-context state. It should minimally track:
    - whether the inventory is open
    - the selected category row
    - the selected item row
    - the currently targeted hand for equip actions
    - a pending action request written by the UI and consumed by the system on the next update

    Create `src/game/systems/InventorySystem.h` and `.cpp` and register the ctx state in `init()`. In `update()`:
    - toggle the menu on `GLFW_KEY_I` per D-14
    - allow `Escape` to close the menu if it is open
    - while open, force `InputSystem::unlockCursor()` and suppress equip/action application if the player entity is absent
    - while closed, restore `InputSystem::lockCursor()` only when closing from the menu itself
    - consume pending equip/unequip actions from `InventoryMenuState` using the `EquipmentState` helper from Plan 01

    Do not reuse `PlayerInteractionLockComponent.active` for inventory pause ownership. That lock is already shared with door flow and is only a flat bool/timer. Instead, make `InventoryMenuState.open` the authoritative pause signal for this feature and update the existing gameplay readers to respect it:
    - `PlayerMovementSystem.cpp` must skip WASD/jump input and zero horizontal intent when inventory is open
    - `CameraSystem.cpp` must stop mouse look when inventory is open
    - `InteractionSystem.cpp` must not request activations while inventory is open

    This phase may still read the existing door/player lock state in those systems, but inventory pause must be additive and separate so closing the menu never clears a door-imposed lock.

    Use the clean two-handed rule from the planning notes: equipping a two-handed weapon into either hand clears conflicting one-handed occupancy and makes both hands report that same weapon as occupied. Do not add quick-cycling, bag slots, non-weapon items, or stat penalties in this phase.

    Keep the system narrow: it owns pause state and action application, not menu rendering. Menu drawing stays in the existing ImGui/render path.
  </action>
