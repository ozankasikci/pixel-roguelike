# Phase 02.1: Equipment Inventory and Carry Weight - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Introduce a Dark Souls-inspired weapon inventory and equipment flow between exploration and combat. This phase defines how weapons are owned, stored, equipped, and presented to the player, using a paused inventory screen and an equip-load-style burden system. It does not expand into a general-purpose loot RPG inventory, non-weapon item management, or broader stat systems.

</domain>

<decisions>
## Implementation Decisions

### Inventory Scope
- **D-01:** This phase covers weapons only. Other carryable item categories are deferred.
- **D-02:** Weapons are inventory items that can be owned independently of being equipped.
- **D-03:** Non-weapon carryables, key items, and broader progression inventory remain out of scope for this phase.

### Equipment Model
- **D-04:** Use an explicit hand-based equipment model inspired by Souls games rather than a single active weapon slot.
- **D-05:** The player has one right-hand weapon slot and one left-hand weapon slot.
- **D-06:** Two-handed weapons are supported as a weapon type and should occupy the hand-equipment model appropriately rather than being treated as a separate unrelated system.
- **D-07:** Weapon changes happen from the inventory screen only for now; in-world quick cycling is deferred.

### Burden / Carry Weight
- **D-08:** Use true equip-load style burden for this phase: equipped weapons determine burden.
- **D-09:** Unequipped weapons in the inventory do not add burden.
- **D-10:** No separate slot-grid or bag-cap inventory model should be introduced in this phase.

### Inventory Presentation
- **D-11:** The inventory is a dedicated paused screen, not a live overlay while gameplay continues.
- **D-12:** The presentation should feel Dark Souls-inspired: category-driven, list-based, and utilitarian rather than grid-based.
- **D-13:** The paused screen should include a category list, item list, detail pane, and a visible burden/equip-load meter.
- **D-14:** The default key to open the inventory is `I`.

### the agent's Discretion
- Exact category names and menu wording, as long as the UI remains Souls-like and minimal
- Exact visual composition, spacing, typography, and 1-bit treatment of the paused inventory screen
- Exact behavior of how a two-handed weapon displaces or locks left/right-hand equipment, as long as it stays consistent with the chosen hand-based model
- Internal data structures for inventory entries, equipment state, and burden calculation

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `RunSession::loadout` in `src/game/session/RunSession.h`: existing place to evolve persistent equipment state
- `ContentRegistry` in `src/game/content/ContentRegistry.h`: existing registry for weapon/item definitions that can absorb weapon weight, handedness, and equip metadata
- `ImGuiLayer` in `src/engine/ui/ImGuiLayer.cpp`: existing immediate-mode UI path for overlays and debug views, useful for early inventory UI iterations

### Established Patterns
- ECS + system-driven game architecture from Phase 01.1 should remain the integration model
- Player-centric interaction currently routes through input polling plus registry context/state rather than monolithic player objects
- World interaction already reserves `E`, and input/cursor state is centrally owned by `InputSystem`

### Integration Points
- Inventory/equipment state should connect to `RunSession` and/or player-related ECS state instead of inventing a disconnected persistence path
- Weapon metadata should likely extend content definitions/assets rather than being hardcoded in systems
- The paused inventory screen must coordinate with cursor-lock/input behavior in `InputSystem`
- Equipped weapon choices will eventually feed combat/viewmodel behavior, so the phase should leave a clean bridge into Phase 3 combat work

</code_context>

<specifics>
## Specific Ideas

- "Weapons should be inventory items."
- "There is no inventory slots."
- "Dark Souls-style" means a list/menu presentation rather than a slot grid.
- The chosen burden model is the Dark Souls-style equip-load interpretation, not a bag-weight simulation.
- The player wants one right hand, one left hand, and support for two-handed weapons.
- For now, weapon switching should happen from the inventory only.

</specifics>

<deferred>
## Deferred Ideas

- General non-weapon inventory management
- Key/progression item handling
- In-world quick cycling of equipped weapon slots
- Broader RPG encumbrance/stat systems beyond equipped-weapon burden

</deferred>
